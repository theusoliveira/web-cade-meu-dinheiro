# Refatoração de performance e manutenção

## O que mudou

- O cálculo do saldo carregado para o mês seguinte foi movido para uma camada dedicada em `lib/finance/balance.ts`.
- A aplicação agora tenta usar a RPC `get_opening_balance(month_start)` no Supabase para trazer somente o número do saldo inicial dos lançamentos pessoais.
- A aba **Lançamentos PJ** reutiliza a mesma estrutura da aba de lançamentos, mas usa tabelas próprias: `pj_entries` e `pj_fixed_entries`.
- A aba PJ também usa uma RPC própria, `get_pj_opening_balance(month_start)`, para carregar o saldo do mês anterior sem misturar dados pessoais.
- Se a RPC ainda não existir no Supabase, o app faz fallback para o cálculo antigo no cliente e continua funcionando.
- Queries do Supabase foram movidas para `lib/supabase/queries/*`.
- O estado de lançamentos, cartão e perfil foi movido para hooks em `hooks/*`.
- As abas principais e os gráficos agora usam `next/dynamic` para reduzir o JavaScript carregado inicialmente.
- O service worker não grava mais páginas navegadas/autenticadas no cache de runtime.

## Ações recomendadas no Supabase

Execute os scripts abaixo pelo SQL Editor do Supabase, colando o conteúdo de cada arquivo:

```txt
supabase/migrations/202604280001_opening_balance.sql
supabase/migrations/202604280002_pj_entries.sql
```

O primeiro otimiza o saldo automático dos lançamentos pessoais. O segundo cria a estrutura independente da aba **Lançamentos PJ**.

## Variáveis de ambiente

O client aceita tanto:

```env
NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY=...
```

quanto a variável antiga:

```env
NEXT_PUBLIC_SUPABASE_ANON_KEY=...
```
