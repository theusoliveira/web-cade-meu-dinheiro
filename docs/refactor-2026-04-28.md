# Refatoração de performance e manutenção

## O que mudou

- O cálculo do saldo carregado para o mês seguinte foi movido para uma camada dedicada em `lib/finance/balance.ts`.
- A aplicação agora tenta usar a RPC `get_opening_balance(month_start)` no Supabase para trazer somente o número do saldo inicial.
- Se a RPC ainda não existir no Supabase, o app faz fallback para o cálculo antigo no cliente e continua funcionando.
- Queries do Supabase foram movidas para `lib/supabase/queries/*`.
- O estado de lançamentos, cartão e perfil foi movido para hooks em `hooks/*`.
- As abas principais e os gráficos agora usam `next/dynamic` para reduzir o JavaScript carregado inicialmente.
- O service worker não grava mais páginas navegadas/autenticadas no cache de runtime.

## Ação recomendada no Supabase

Execute o script:

```sql
supabase/migrations/202604280001_opening_balance.sql
```

Sem esse script, o app continua funcionando usando fallback, mas sem o ganho principal de performance no cálculo do saldo anterior.

## Variáveis de ambiente

O client aceita tanto:

```env
NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY=...
```

quanto a variável antiga:

```env
NEXT_PUBLIC_SUPABASE_ANON_KEY=...
```
